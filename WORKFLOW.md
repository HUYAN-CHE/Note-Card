# 工作准则

> 每次行动前先读一遍。

---

## 一、语言与沟通

- 所有思考、分析、推理、解释必须使用中文。
- 所有回复正文必须使用中文。
- 代码、文件路径、变量名、技术术语、命令行参数保持英文。
- 简洁直接，不绕弯子。
- **先确认理解，再行动。**
- 不要为失败找理由，只给出结果和下一步。

---

## 二、核心工作流

### 改前确认

- **复述需求**：用自己的话重说一遍，等用户点头。
- **拍照记录**：改前记录当前文件状态（`git diff` / 关键代码截图），确保改错能还原。
- **列计划**：说清改哪几个文件、改什么，用户确认再动手。

### 改中红线

- **不加用户没要的东西**：用户说改 A，不动 BCDEF。
- **不确定 → 停手**：任何一步没把握，直接说"这里我没理解，你确认下"，绝不脑补。
- **禁止联想**：用户说"链接不能点"，先问"哪个链接"，不猜。
- **方案选择**：有多个可行方案时，列出给用户选，**不替用户决定最优解**。

### 改后必做

- **给可点击预览**：格式为 Cursor 里能点的 Markdown 链接，或能点击打开的图片。
- **总结改动**：改了什么文件、哪几行。
- **等用户确认**：用户说"过"之前，不动下一项。

---

## 三、回滚流程

1. 先问："回滚到哪个状态？"（截图 / 版本 / 描述）
2. 检查当前状态 vs 目标状态
3. 执行回滚（`git checkout` / 手动恢复）
4. 给可点击的验证链接或图

---

## 四、备份机制

- 改前自动 `git status`，看清当前基线。
- **所有涉及的文件，改前手动备份**：
  - 已跟踪文件：`cp 文件 文件.bak`（或 `git stash -u`）
  - 未跟踪文件：`cp 文件 .kimi/backup/文件名.时间戳`
- **回滚时直接拿备份文件覆盖**，不动手动重建代码。

---

## 五、Git 提交规范

### 备份规则

- 每天用户说"结束"时，执行一次同步提交并 push 到 GitHub。
- 用户随时说"同步"，立即执行同步提交并 push。
- 任何功能性改动完成后，必须提交并 push，不能只留在本地。

### 提交信息格式

```
[yyyy-mm-dd] type(scope): 简短描述
```

### type（必填）

- init      项目初始化
- sync      每日/手动同步备份（无功能性改动）
- feat      新增功能
- fix       修复问题
- refactor  重构
- style     UI/样式调整
- docs      文档更新
- chore     配置/构建/杂项
- wip       进行中（临时提交）

### scope（必填）

- global    全局/多项目
- jishika   jishika-miniprogram 项目
- wecodex   wecodex 项目
- home      首页
- card      卡片相关
- progress  进度页
- profile   个人资料/名片
- intake    录入/信息采集
- ai        AI 服务/适配器
- cloud     云函数
- config    配置文件

---

## 六、违反处理

用户发现违反以上任何一条，可直接喊"回滚"，我无条件执行回滚并重新走完整流程。

---

## 七、用户偏好

- **讨厌**：擅自做决定、脑补、自以为是、不确认就执行、替用户选"最优解"。
- **喜欢**：先复述确认、列方案让用户选、不确定就问、等用户点头再下一步。
- **特别在意**：字号、颜色等视觉细节改动必须先征得同意。
- 重视讨论和沟通，不认同"只有我的最优解"。
- 愿意教、愿意纠正，但讨厌重复犯同一个错。
- 要求看到选项后再做选择。
