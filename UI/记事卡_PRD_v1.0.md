# 《记事卡》产品需求文档（PRD）

> 版本：v1.0 | 作者：产品团队 | 日期：2026-07-14
> 状态：定稿

---

## 1. 概述

### 1.1 背景与动机

《记事卡》是一个"把一件事说清楚、找对人来帮忙"的互助协作工具。用户把待办/需求写成一张"记事卡"，通过一度人脉（好友）或直接邀请，把卡推给能帮上忙的人；好友看到卡后既能被动浏览二度人脉，也能主动"介绍给朋友"——把卡连同自己的关系链转给更匹配的人，形成可溯源的"互助网络"。

核心洞察：在微信生态中，大量协作需求散落在聊天里——帮朋友找资源、约人、询问信息、协调事务。用户不想拉群、不想在聊天里反复翻。《记事卡》介于"待办清单"与"熟人协作网络"之间：比待办多了**人脉协作**，比社群轻了**建群负担**。

### 1.2 目标

- **业务目标**：打造微信小程序内闭环的轻量协作工具，通过关系链扩散形成网络效应
- **用户目标**：让"一件事"和"能解决它的人"高效连接，降低协作门槛
- **成功指标**：
  - 北极星指标："聊到→结构化草稿就绪"从分钟级降到秒级
  - 护栏指标：邀请接受率、单卡协作完成率、二度扩散率
  - 成本指标：单卡 AI 调用成本压到免费额度内

### 1.3 范围

- **本期包含（MVP）**：
  - 7 个主页面：事项（首页）、互助、我的主页、记事卡详情、新建/编辑记事卡、邀请好友、从聊天整理
  - 核心功能：记事卡 CRUD、协助者邀请、二度人脉浏览与申请、从聊天一键整理（A+C方案）、订阅消息通知
  - AI 能力（P0）：混元语义结构化、智能协助者推荐
- **本期不包含**：
  - 群协作（多创作者）
  - 付费/积分体系
  - 跨平台（仅微信小程序）
  - 实时聊天（依赖微信原生）

## 2. 用户画像

### 2.1 角色定义

| 角色 | 身份 | 核心目标 | 主要痛点 |
|------|------|----------|----------|
| **创作者**（如：胡晓） | 发起记事卡的人 | 把聊天里的事变成卡，邀请能帮的人，跟进进度到办成 | 信息散落在聊天里，无沉淀；不确定该找谁帮忙 |
| **一度协助者**（如：王姐） | 被直接邀请的好友 | 收到邀请后接受，参与推进待确认，完成协作 | 协作进度不透明，不知道要做什么 |
| **二度访客**（如：小林） | 好友的好友 | 浏览扩散来的卡，找到能帮上忙的事申请加入 | 不知道朋友的朋友需要什么，缺少连接渠道 |

### 2.2 用户旅程地图（创作者主线）

| 阶段 | 情绪 | 动作 | 触点 | 痛点 | 机会 |
|------|------|------|------|------|------|
| 1·触发 | 😟 | 微信聊到一件事，怕忘 | 微信会话 | 信息散落，无沉淀 | 一键从聊天整理 |
| 2·创建 | 😀 | A转发/C复制→预览→生成 | 微信AI+小程序 | 不想从头填表单 | 自动识别+可勾改 |
| 3·发布邀请 | 🙂 | 填协助者/可见性→发布 | 小程序+订阅消息 | 不确定该找谁 | 聊天识别协助者候选 |
| 4·协作中 | 😌 | 协助者接受、推进进度 | 小程序内协作 | 进度不透明 | 状态双编码+进度 |
| 5·二度扩散 | 🤔 | 申请加入需审批 | 互助页+审批 | 审批打断心流 | 一键通过+订阅提醒 |
| 6·完成 | 😄 | 事办成，标记完成 | 小程序闭环 | 办完无正反馈 | 完成庆祝+信任分 |

## 3. 用户故事

### 3.1 创作者（Creator）

**US-C01**：作为创作者，我想要从微信聊天一键生成记事卡，以便不用在小程序内从头填写。
**US-C02**：作为创作者，我想要邀请微信好友当协助者，以便把卡推给能帮上忙的人。
**US-C03**：作为创作者，我想要设置卡的可见性（仅协助者/二度可看），以便控制隐私范围。
**US-C04**：作为创作者，我想要审批二度访客的加入申请，以便确保协作质量。
**US-C05**：作为创作者，我想要看到协助者的进度更新，以便掌握协作状态。
**US-C06**：作为创作者，我想要标记记事卡完成，以便沉淀信任关系。

### 3.2 一度协助者（1° Helper）

**US-H01**：作为一度协助者，我想要收到订阅消息通知，以便及时知道被邀请协助。
**US-H02**：作为一度协助者，我想要查看记事卡详情和待确认事项，以便了解要做什么。
**US-H03**：作为一度协助者，我想要更新进度和补充信息，以便推进协作。
**US-H04**：作为一度协助者，我想要"介绍给朋友"把卡转给更合适的人，以便扩大匹配范围。

### 3.3 二度访客（2° Visitor）

**US-V01**：作为二度访客，我想要在互助页浏览好友的朋友的需求，以便找到能帮上忙的事。
**US-V02**：作为二度访客，我想要申请加入感兴趣的记事卡，以便参与协作。
**US-V03**：作为二度访客，我想要看到申请审批状态，以便知道是否被接受。

## 4. 功能需求

### 4.1 功能清单

#### 核心功能（Must Have）

| 编号 | 功能名称 | 用户故事 | 功能描述 | 输入/输出/交互 |
|------|----------|----------|----------|----------------|
| F-001 | 事项首页 | US-C01 | 我创建的/与我相关的记事卡列表；"从聊天整理"快捷入口；FAB新建 | 列表数据、筛选状态、点击跳转 |
| F-002 | 从聊天整理 | US-C01 | A路径：转发微信AI结构化；C路径：剪贴板识别；②路径：选文件 | 聊天文本/图片→结构化草稿→预览页 |
| F-003 | 整理预览 | US-C01 | 标题/描述/待确认/协助者候选四块，默认全勾选，可取消编辑 | 结构化数据→用户确认→生成卡片 |
| F-004 | 新建/编辑记事卡 | US-C01,US-C03 | 标题→描述→待确认→协助者→可见性开关的线性分组 | 表单输入→保存草稿/发布 |
| F-005 | 记事卡详情 | US-C05 | 详情+协助者+待确认+进度；三种身份视图（创作者/一度/二度） | 卡ID→按身份渲染不同视图 |
| F-006 | 邀请好友 | US-C02 | 底部sheet多选好友+群聊入口；发送订阅消息 | 选择好友→发送邀请→建立Relationship |
| F-007 | 互助页 | US-V01 | 一度人脉横滑→点击刷新下方二度人脉卡片；空态引导 | 一度列表→二度卡列表→点击进入详情 |
| F-008 | 申请加入 | US-V02 | 二度访客填写说明提交JoinRequest，创作者审批 | 申请表单→审批流程→建立关系 |
| F-009 | 介绍给朋友 | US-H04 | 一度/二度viewer把卡连同身份转发给好友，带introducedBy链路 | 选择好友→生成分享卡→好友申请带链路 |
| F-010 | 我的主页 | US-C06 | 个人资料+创建的/协助的/收藏子标签+互助统计 | 用户ID→统计数字+列表 |
| F-011 | 订阅消息 | US-H01 | 邀请通知、申请通知、进度更新通知，通过微信云调用发送 | 事件触发→订阅消息→用户点击 |

#### AI 功能（Should Have）

| 编号 | 功能名称 | 用户故事 | 功能描述 | 技术方案 |
|------|----------|----------|----------|----------|
| F-012 | AI语义结构化 | US-C01 | 混元抽取标题/描述/待确认/协助者候选 | aiStructure云函数 |
| F-013 | 智能协助者推荐 | US-C02 | 根据卡需求与好友画像排出最该邀请的3人 | aiRecommend云函数 |
| F-014 | 进度摘要 | US-C05 | 定时聚合某卡的协作更新，混元压成进度摘要 | aiSummary云函数 |

### 4.2 非功能需求

| 维度 | 要求 |
|------|------|
| **性能** | 首屏加载 < 2s；列表滚动 60fps；AI结构化响应 < 3s |
| **安全** | 混元API Key仅存云开发环境变量，端上不暴露；调用走云函数 |
| **可用性** | 微信小程序原生体验；正文14px起；触控目标≥44px |
| **可扩展性** | 支持关系链度数上限可配置（默认3度）；数据模型预留字段 |
| **易用性** | 浅/深双主题；状态文字+圆点双编码；教育性空状态 |
| **兼容性** | 微信小程序基础库 2.30+；iOS/Android 兼容 |

### 4.3 功能依赖关系

```
F-002（从聊天整理）→ F-003（整理预览）→ F-004（新建/编辑）→ F-005（详情）
                                                          ↓
F-006（邀请好友）←→ F-011（订阅消息）→ US-H01（接受邀请）→ F-005（协作）
                                                          ↓
F-007（互助页）← F-004（可见性开关开启）
  ↓
F-008（申请加入）→ F-011（审批通知）→ F-005（创作者审批视图）
  ↓
F-009（介绍给朋友）→ F-008（申请加入，带introducedBy）
```

## 5. 优先级

### 5.1 MoSCoW 矩阵

| 编号 | 功能名称 | 优先级 | 理由 |
|------|----------|--------|------|
| F-001 | 事项首页 | **Must** | 核心入口，没有无法使用 |
| F-002 | 从聊天整理 | **Must** | 核心差异化功能，降低创建门槛 |
| F-003 | 整理预览 | **Must** | F-002的必要后续 |
| F-004 | 新建/编辑记事卡 | **Must** | 核心创建流程 |
| F-005 | 记事卡详情 | **Must** | 核心查看/协作页面 |
| F-006 | 邀请好友 | **Must** | 协作关系建立的关键 |
| F-007 | 互助页 | **Must** | 二度扩散的核心入口 |
| F-008 | 申请加入 | **Must** | 二度协作闭环 |
| F-009 | 介绍给朋友 | **Must** | 关系链转发的核心 |
| F-010 | 我的主页 | **Must** | 个人中心+统计 |
| F-011 | 订阅消息 | **Must** | 通知闭环 |
| F-012 | AI语义结构化 | **Should** | 体验升级，本地规则可兜底 |
| F-013 | 智能协助者推荐 | **Should** | 邀请体验优化 |
| F-014 | 进度摘要 | **Could** | 体验增厚，非必需 |
| F-015 | 语音建卡 | **Won't** | 本期不做，留待后续 |
| F-016 | 信任分模型 | **Won't** | 数据积累后再做 |

**占比统计**：Must 11/16（69%）| Should 2/16（12%）| Could 1/16（6%）| Won't 2/16（13%）

> Must 占比略超60%，因MVP阶段核心功能必须全部上线。AI功能由Should/Could承担，确保核心流程不受影响。

### 5.2 版本规划建议

- **MVP（v1.0）**：F-001~F-011（所有 Must Have）
- **v1.1**：F-012 + F-013（AI 能力上线）
- **v2.0**：F-014 + 评估信任分模型

## 6. 验收标准

### 6.1 F-001 事项首页

**AC-F001-01**：正常加载
```
Given 用户已登录且创建了至少1张记事卡
When 打开事项首页
Then 按更新时间倒序展示记事卡列表，每张卡显示标题+状态+协助者头像+更新时间
```

**AC-F001-02**：空状态引导
```
Given 用户未创建任何记事卡
When 打开事项首页
Then 展示教育性空状态：插画+"还没有记事卡"+"下拉或点击上方按钮，从聊天记录生成一张记事卡"
```

**AC-F001-03**：快捷入口
```
Given 用户在事项首页
When 点击"从聊天整理"快捷入口
Then 跳转到从聊天整理页（/pages/card-import）
```

### 6.2 F-002 从聊天整理

**AC-F002-01**：剪贴板识别（C路径）
```
Given 用户已复制微信聊天内容
When 打开从聊天整理页
Then 自动检测剪贴板内容，显示"检测到剪贴板内容"卡片，点击"粘贴并整理"进入预览
```

**AC-F002-02**：空剪贴板处理
```
Given 用户剪贴板为空
When 打开从聊天整理页
Then 显示引导提示"去微信复制聊天内容，或选择图片/文件"，提供选文件入口
```

**AC-F002-03**：隐私声明
```
Given 用户在从聊天整理页
Then 页面顶部显示隐私声明："记事卡不会主动读取微信消息，以下内容都需你主动复制或选择"
```

### 6.3 F-003 整理预览

**AC-F003-01**：字段展示与勾选
```
Given 用户已完成剪贴板粘贴或AI结构化
When 进入整理预览页
Then 展示标题/描述/待确认/协助者候选四个区块，默认全部勾选，可取消勾选或编辑内容
```

**AC-F003-02**：生成记事卡
```
Given 用户在整理预览页已勾选至少一个字段
When 点击"生成记事卡"
Then 跳转到新建记事卡页，已勾选字段自动预填，未勾选字段留空
```

### 6.4 F-004 新建/编辑记事卡

**AC-F004-01**：表单填写
```
Given 用户在新建记事卡页
When 填写标题（必填）、描述、待确认事项、协助者、可见性开关
Then 可以保存为草稿或点击"邀请一起用"发布
```

**AC-F004-02**：可见性开关
```
Given 用户在新建记事卡页
When 开启"让朋友的朋友也能看到"
Then 该卡发布后可在互助页被二度人脉浏览；关闭则仅协助者可见
```

**AC-F004-03**：标题校验
```
Given 用户未填写标题
When 点击发布
Then 阻止提交，提示"请输入标题"
```

### 6.5 F-005 记事卡详情

**AC-F005-01**：创作者视图
```
Given 用户是卡的创作者
When 进入记事卡详情
Then 显示完整编辑权限：可编辑内容、可邀请、可审批申请、可标记完成
```

**AC-F005-02**：一度协助者视图
```
Given 用户是卡的一度协助者
When 进入记事卡详情
Then 显示协作视图：可查看全部内容、可更新待确认、不可编辑标题/描述
```

**AC-F005-03**：二度访客视图
```
Given 用户是二度访客
When 进入记事卡详情
Then 显示只读视图：可查看标题/描述/进度，不可编辑，底部显示"申请加入"按钮
```

### 6.6 F-006 邀请好友

**AC-F006-01**：好友选择
```
Given 用户在记事卡详情页点击"邀请"
When 底部sheet弹出
Then 显示一度人脉列表，可多选，点击好友头像切换选中状态
```

**AC-F006-02**：发送邀请
```
Given 用户已选择至少1位好友
When 点击"发送给好友"
Then 调用订阅消息API，好友收到"XX邀请你协助《XXX》"通知，Relationship写入一度关系
```

### 6.7 F-007 互助页

**AC-F007-01**：一度人脉展示
```
Given 用户有至少1位一度人脉
When 打开互助页
Then 顶部横向滚动展示协作过的人头像，点击选中后下方刷新该一度下的二度卡列表
```

**AC-F007-02**：空状态
```
Given 用户没有一度人脉
When 打开互助页
Then 显示教育性空状态："还没有一起协作过的人"+"邀请微信好友一起用记事卡"+邀请按钮
```

### 6.8 F-008 申请加入

**AC-F008-01**：申请提交
```
Given 二度访客在记事卡详情页
When 点击"申请加入"
Then 弹出底部sheet，填写申请说明，提交JoinRequest，状态为"待审"
```

**AC-F008-02**：创作者审批
```
Given 创作者收到申请通知
When 进入记事卡详情→审批区域
Then 显示申请人信息+申请说明，可选择"通过"或"拒绝"
```

**AC-F008-03**：通过后权限升级
```
Given 创作者通过了申请
When 申请被批准
Then 申请人成为该卡的一度协助者（degree=2），可参与协作
```

### 6.9 F-009 介绍给朋友

**AC-F009-01**：介绍流程
```
Given 一度/二度协助者在记事卡详情
When 点击"介绍给朋友"
Then 选择好友，生成带"由XX介绍"的分享卡片，好友打开后申请自动带introducedBy和链路
```

### 6.10 F-010 我的主页

**AC-F010-01**：统计展示
```
Given 用户进入我的主页
Then 显示：协作过的人（12）、被求助次数（8）、创建的卡数、协助的卡数
```

**AC-F010-02**：子标签切换
```
Given 用户在我的主页
When 切换"未完成"/"已完成"/"我协助的"标签
Then 下方列表按条件筛选展示对应记事卡
```

### 6.11 F-011 订阅消息

**AC-F011-01**：邀请通知
```
Given 用户A邀请用户B协助记事卡
When 邀请发送成功
Then 用户B收到订阅消息："XX邀请你协助《XXX》"，点击进入记事卡详情
```

**AC-F011-02**：申请通知
```
Given 二度访客提交了加入申请
When 申请提交成功
Then 创作者收到订阅消息："XX申请加入《XXX》"，点击进入审批视图
```

## 7. 数据模型

### 7.1 实体关系

| 模型 | 关键字段 | 说明 |
|------|----------|------|
| **User** | _openid, name, avatar, trustScore | 每个用户唯一标识 |
| **Relationship** | from, to, degree(1\|2\|3), source, introducedBy?, introducedPath[] | 一度=直接好友；二度=好友的好友 |
| **Card** | _id, creatorId, title, desc, todos[], helpers[], isNetworkVisible, maxIntroduceDepth, status, createdAt | 记事卡本体 |
| **JoinRequest** | cardId, applicantId, degree, introducedBy?, introducedPath[]?, status, appliedAt | 申请加入凭证 |

## 8. 假设与约束

### 8.1 假设

- 用户已熟悉微信小程序的基本操作（下拉、分享、订阅消息授权）
- 微信 AI（元宝）已上线聊天记录总结能力，可作为A路径的依赖
- 用户愿意授权剪贴板读取（iOS需手势触发）
- 云开发免费额度足够支撑MVP阶段的AI调用量
- 关系链扩散默认上限3度，可防止信任稀释

### 8.2 约束

- **技术约束**：小程序无法主动读取聊天记录（微信隐私红线）；普通3P小程序无API接收被转发的聊天文本
- **平台约束**：仅微信小程序；依赖微信订阅消息模板
- **成本约束**：混元Token免费额度有限，需设计成本护栏（先本地规则、再AI兜底、结果缓存）
- **合规约束**：内容安全需经用户主动复制/选择后才进入AI处理流程

### 8.3 风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 微信AI（元宝）接口不稳定 | 高 | 中 | C路径（剪贴板）始终可用，确保核心流程不中断 |
| 用户不理解二度人脉概念 | 中 | 中 | 教育性空状态+引导文案+视觉分层 |
| 订阅消息授权率低 | 中 | 中 | 在关键节点（邀请/申请）引导授权，说明价值 |
| 混元Token超支 | 高 | 低 | 成本护栏：本地规则优先、结果缓存、小模型配置 |
| 关系链扩散失控 | 中 | 低 | 度数上限3度+maxIntroduceDepth可配置 |
