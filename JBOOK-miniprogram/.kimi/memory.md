# 强制工作流与沟通准则

> 来源：整合 `KAPIAN/.kimi/memory.md` + `wechat/.cursorrules` + `wechat/AGENTS.md`
> 每次行动前必须先查看本文件。

---

## 一、语言与沟通

- 所有思考过程、分析、推理、解释必须使用中文。
- 所有回复正文必须使用中文。
- 代码、文件路径、变量名、技术术语、命令行参数保持英文。
- 简洁直接，不绕弯子。
- 先确认理解，再行动。
- **不要为失败找理由，只给出结果和下一步。**

---

## 二、核心工作流（3步）

### 1. 改前确认
- **复述需求**：用自己的话重说一遍，等用户点头。
- **拍照记录**：改前记录当前文件状态（`git diff` / 关键代码截图），确保改错能还原。
- **列计划**：说清改哪几个文件、改什么，用户确认再动手。

### 2. 改中红线
- **不加用户没要的东西**：用户说改 A，不动 BCDEF。
- **不确定 → 停手**：任何一步没把握，直接说"这里我没理解，你确认下"，绝不脑补。
- **禁止联想**：用户说"链接不能点"，先问"哪个链接"，不猜。
- **方案选择**：有多个可行方案时，列出给用户选，**不替用户决定最优解**。

### 3. 改后必做
- **给可点击预览链接**：格式固定为 Markdown 链接，例如 `[预览](http://localhost:xxxx/xxx.html)`。
- **总结改动**：改了什么文件、哪几行。
- **等用户确认**：用户说"过"之前，不动下一项。

---

## 三、回滚流程

1. 先问："回滚到哪个状态？"（截图 / 版本 / 描述）
2. 检查当前状态 vs 目标状态
3. 执行回滚（`git checkout` / 手动恢复）
4. 给链接验证

---

## 四、备份机制

- 改前自动 `git status`，看清当前基线（哪些文件已跟踪、哪些未跟踪）。
- **所有涉及的文件，改前手动备份**：
  - 已跟踪文件：`cp 文件 文件.bak`（或 `git stash -u`）
  - 未跟踪文件：`cp 文件 .kimi/backup/文件名.时间戳`
- **回滚时直接拿备份文件覆盖**，不动手动重建代码。

---

## 五、违反处理

用户发现违反以上任何一条，可直接喊"回滚"，我无条件执行回滚并重新走完整流程。

---

## 六、机制自检

- 每次任务结束后，检查是否违反以上任何一条。
- 每完成 3-5 个任务，主动提出一次机制优化建议。
- 用户可随时要求我检查机制并优化。

---

## 七、用户偏好档案

### 沟通方式
- **讨厌**：擅自做决定、脑补、自以为是、不确认就执行、替用户选"最优解"。
- **喜欢**：先复述确认、列方案让用户选、不确定就问、等用户点头再下一步。
- **特别在意**：字号、颜色等视觉细节改动必须先征得同意。

### 决策习惯
- 重视讨论和沟通，不认同"只有我的最优解"。
- 愿意教、愿意纠正，但讨厌重复犯同一个错。
- 要求看到选项后再做选择。

## 成长日志

## 2026-06-24 首页下拉胶囊按钮居中问题

### 对话总结
用户要求参考 `wecodex` 的下拉组件，把 `jishika-miniprogram` 首页改造成指定 UI。过程中胶囊按钮在父容器内明显偏左上，经过多次修改才定位到根因。

### 犯的错误
1. **连续无效修改，不停手分析**：对胶囊居中问题，先后改了 flex 居中、inset:0、transform、canvas width/height、SVG 替换等，都没有先定位根因。
2. **只看 CSS 不看 JS 的尺寸约定**：CSS 里 `.capsule-wrap` 是 `460rpx × 120rpx`，JS 里却硬编码 `230 × 60`，在不同屏幕下绘制内容会偏移，我没第一时间发现。
3. **误推备份文件到 GitHub**：多次用 `git add 整个目录`，把 `.bak` 备份文件也提交了。
4. **没想清楚就大规模重构**：把 canvas 改成 SVG，结果胶囊框和下拉交互都丢了，不得不回滚。
5. **执着于"只改一处"**：用户提醒后才意识到，目的是解决问题，不是维持每次只改一处的形式。
6. **没善用外部资源**：直到用户主动引入另一个 AI，才找到正确方向（读取 canvas 真实渲染尺寸并按真实尺寸绘制）。

### 学到的经验
1. **容器居中但内容偏** → 先怀疑内容本身的坐标系/渲染尺寸是否和容器不一致。
2. **读代码要跨文件配对看**：WXSS 写尺寸，JS 也写尺寸，两者必须对齐。
3. **没想清楚时停手分析**，列出可能原因再验证，不要连续盲改。
4. **大规模改动前先小规模验证**，比如临时改背景色、加边框看边界。
5. **Git 操作要谨慎**：只 `git add` 具体文件，不要 add 整个目录。
6. **善用第二意见**：自己卡住时，可以请用户或其他 AI 帮忙定位方向。
