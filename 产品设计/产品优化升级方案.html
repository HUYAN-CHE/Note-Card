03

云函数设计

入参 / 出参 / 与数据模型的关系。
aiStructure
P0① · 聊天→结构化草稿
入参为剪贴板/截图 OCR 文本（来自 从聊天整理 C 路径或 ②选文件），调用混元抽取结构；结果写 Card 草稿，建议协助者候选回写 Relationship/User 查询。

▶ 入参（event）

rawText: string // 复制的聊天/截图OCR source: 'clipboard' | 'image' | 'forward' cardDraftId?: string // 已有草稿则补写 openid: string // 创建者

◀ 出参（return）

title: string desc: string todos: [{ text, due? }] budget?: number helperCandidates: [openid] // 聊天提及 confidence: 0..1
aiRecommend
P0② · 最该邀请谁
读取 Card + Relationship 一度/二度图 + User 画像，混元语义匹配"需求 × 人"，输出带理由的候选列表。仅在用户主动建卡/邀请时调用。

▶ 入参（event）

cardId: string degree: 1 | 2 // 默认一度 limit: number // 默认 3 openid: string

◀ 出参（return）

candidates: [{ userId, score: 0..1, reason: string // 可解释 }] fromCache: boolean
aiSummary
P1③ · 进度摘要
定时（云函数定时器）聚合某卡的协作更新，混元压成 1 条待确认 + 进度摘要，写回 Card.todos / 进度字段，降低"进度不透明"。

▶ 入参（event）

cardId: string since: timestamp // 上次摘要时间

◀ 出参（return）

summary: string pendingUpdates: [{ text, by: openid }] doneCount: number
数据模型关系 4 模型
1
aiStructure 写入
Card(title/desc/todos/budget) 草稿；helperCandidates 用于查询 Relationship/User
2
aiRecommend 读取
Card + Relationship(degree) + User(画像) → 输出候选；用户确认后写 Relationship(一度)
3
aiSummary 回写
聚合更新 → 写 Card.todos/进度；二度申请经 JoinRequest 审批后升级权限

密钥：混元 API Key 仅存云开发环境变量，端上不暴露；调用走云函数，符合隐私合规（内容经用户复制/选择后才进入）。

🛡️ 免费 Token 成本护栏（落地原则）
  • 只在用户主动粘贴/建卡/邀请时调用，不做后台常驻扫描聊天（也不合规）。
  • 先本地规则、兜底再 AI：正则能搞定的不烧 Token，识别不准才走混元。
  • 结果缓存：同一段剪贴板哈希缓存，重复建卡不重复计费。
  • 小模型 / 低 max_tokens：抽取类任务用最小可用配置。
  • 二度匹配批处理：夜间离线跑，不在请求链路实时调。
